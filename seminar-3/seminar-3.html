InitializationTest

Что здесь происходит?

  • Класс должен быть проинициализирован, прежде чем его экземпляры могут быть созданы:

    • Это происходит только один раз

    • После этого можно использовать класс

Процесс инициализации класса

Включает в себя:

  1. Инициализацию суперклассов (родителей)

  2. Инициализацию static полей

  3. Вызов блоков static инициализации (в том же порядке, что и в исходном коде)

Инициализация экземпляра

В JLS это называется "Обработкой вызова конструктора"
Включает в себя (рекурсивно):

  1. Обработка вызова конструктора суперкласса

  2. Вызов блоков инициализации и присваивание полей (в том же порядке, что и в исходном коде)

  3. Вызов конструктора

На практике

public class Main {
  public static class A {
    static { System.out.println("1"); }
    static { System.out.println("2"); }
  }

  public static class B extends A {
    static { System.out.println("3"); }
    static { System.out.println("4"); }
  }

  public static void main(String[] args){
    new B();
  }
}

На практике (2)

public class Main {
  public static class A {
    { System.out.println("1"); }
    A() { System.out.println("3"); }
    { System.out.println("2"); }
  }

  public static class B extends A {
    B() { System.out.println("6"); }
    { System.out.println("4"); }
    { System.out.println("5"); }
  }
}

IntegerConstTest

Кэши обёрток

  • В Java обёртки для примитивов кэшируются Integer, Long, Character, Byte, Boolean, Short

  • И это записано в спецификации

  • Integer, Long, Short, Byte - [-128; 127]

  • Character - [0; 127]

  • Boolean - {true, false}

Кэши обёрток

Пример:

public class IntegerCacheDemo {
  public static void main(String[] args){
    Integer a = 127;
    Integer b = 127;
    Integer c = 128;
    Integer d = 128;
    System.out.println(a == b); // true
    System.out.println(c == d); // false

  }
}

String.intern и прочее

String.intern

Из Javadoc:

  • Возвращает каноничное представление строки

  • То есть для любых строк a и b

    • если a.equals(b) == true

    • то a.intern() == b.intern()

Не рекомендуется использовать (может создать излишнюю нагрузку на внутреннюю таблицу строк)

String.intern

@SuppressWarnings("all")
public class InternTest {
  public static void main(String[] args){
    String a = new String("abc").intern();
    String b = new String("abc").intern();

    System.out.println("equals: " + a.equals(b)); // true
    System.out.println("==: " + a == b); // true
  }
}

String и ==

  • Строки, являющиеся значениями константных выражений интернируются

  • Подробнее - JLS §3.10.5

String и ==

Пример из JLS:

package testPackage;
class Test {
    public static void main(String[] args) {
        String hello = "Hello", lo = "lo";
        System.out.println(hello == "Hello"); // true
        System.out.println(Other.hello == hello); // true
        System.out.println(other.Other.hello == hello); // true
        System.out.println(hello == ("Hel"+"lo")); // true
        System.out.println(hello == ("Hel"+lo)); // false
        System.out.println(hello == ("Hel"+lo).intern()); // true
    }
}
class Other { static String hello = "Hello"; }

finalize, Сборка мусора (GC)

finalize

  • Вызывается, когда GC понимает, что ссылок на объект нет

  • Может использоваться для освобождения некоторых ресурсы

  • Вызывается только один раз

  • Но

finalize

Javadoc: The finalization mechanism is inherently problematic.

finalize

  • Помечен deprecated

  • Не должен использоваться

GCDemo

  • Сборка мусора - деталь реализации

  • Не нужно полагаться на явный вызов System.gc

  • Объекты могут быть удалены даже если на них "есть ссылки"
